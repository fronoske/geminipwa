# ソース分割・ビルド導入計画

## 目的

現在の `index.html` に同居しているHTML、CSS、JavaScriptを、挙動を維持しながら段階的に分離する。
最終的にはHTML、CSS、TypeScriptをソースとして管理し、GitHub Pagesへ公開する成果物をビルドで生成する。

この移行では、IndexedDBに保存済みの設定・会話履歴との互換性と、PWAとしての更新・オフライン動作を最優先する。

## 現状

- `index.html` は約22,000行、約1.26MB。
- 約4,880行のCSSと、約14,000行の主要JavaScriptがインラインで記述されている。
- JavaScriptはグローバルな `state`、`elements`、`dbUtils`、`uiUtils`、`appLogic` を中心に相互参照している。
- IndexedDBのデータベース名は `GeminiPWA_DB`、現在のスキーマバージョンは8。
- `sw.js` は `index.html`、`manifest.json`、`marked.js` などのファイル名を固定してキャッシュしている。
- GitHub Pages向けのビルド・デプロイワークフローはなく、現在はリポジトリ直下の静的ファイルをそのまま公開する構成。
- HTML属性のインラインイベントハンドラーは見当たらず、外部スクリプト化を妨げる要因は比較的少ない。

## 移行原則

1. 各フェーズで、起動可能かつレビュー可能な状態を維持する。
2. 機械的な移動と、機能変更・リファクタリングを同じコミットに混在させない。
3. 移行中はIndexedDB名、ストア名、キー、設定名を変更しない。
4. CSSセレクター、DOM ID、イベント発火順序を、分離だけを行うフェーズでは変更しない。
5. `dist/` は生成物とし、ソースとの差分レビュー対象にしない。
6. 公開方法を切り替えるまでは、既存のルート配置を壊さない。
7. ヘッダーメニュー変更などの機能改修は、ビルド結果が現行版と同等であることを確認した後に行う。

## フェーズ

### 0. ベースラインの固定

- 現行版の主要操作を手動スモークテスト項目として記録する。
- インラインJavaScriptの構文検査を自動化する。
- IndexedDBの設定・会話履歴をエクスポートし、移行前後で再利用できることを確認する。
- 画面幅の代表値を決め、チャット・履歴・設定画面の基準スクリーンショットを取得する。

完了条件: 現行版の合否を判断できるチェックリストがあり、移行後と比較できる。

### 1. ビルド基盤の導入

- Vite、TypeScript、テストランナーを導入する。
- Viteの `base` はGitHub Pagesのサブパスでも動作する相対パス構成にする。
- `npm run dev`、`npm run build`、`npm run typecheck`、`npm test` を定義する。
- この段階ではアプリ本体を分割せず、既存のルート配置も残す。

完了条件: 空の検証用エントリではなく、現行アプリをビルド経由で起動できる。

### 2. 挙動を変えない外部ファイル化

- インラインCSSを `src/styles/app.css` へ移す。
- 起動監視、主要ロジック、入力プリセットをそれぞれ外部ファイルへ移す。
- まずJavaScriptとして同じ実行順序を保ち、差分がファイル移動中心になるようにする。
- CDN依存のDOMPurifyとMermaid、ローカルのMarkedは、この段階では読み込み方式を変えない。

完了条件: DOM、CSS、主要操作、保存データ、外部APIへのリクエスト形式が現行版と同等。

### 3. TypeScriptへの段階移行

- 共通のデータ型から定義する: 設定、メッセージ、チャット、添付、APIプロバイダー。
- DOM要素マップとIndexedDBアクセスに型を付ける。
- `state` の暗黙的な形を型として固定し、設定項目の欠落をコンパイル時に検出できるようにする。
- APIクライアント境界へ要求・応答型を導入する。
- 一時的な型抑制を使う場合はファイル単位で理由を記録し、解消対象を明示する。

完了条件: `npm run typecheck` が成功し、新しい設定キーの追加漏れを型検査で検知できる。

### 4. 機能モジュール化

- 依存の少ない純粋関数、IndexedDB、APIクライアント、UIの順に分割する。
- グローバル変数の直接参照を減らし、必要な状態とDOM要素を引数または限定されたコンテキストで渡す。
- Gemini、DeepSeek、Anthropic、OpenAI互換APIの共通部分と固有部分を整理する。
- チャット、履歴、設定の各画面を独立した初期化単位にする。

完了条件: 主要モジュールに循環依存がなく、単体テスト可能な境界ができている。

### 5. PWAとデプロイの切り替え

- Service WorkerがViteのハッシュ付き成果物を正しく更新・キャッシュできる方式へ変更する。
- GitHub Actionsで `dist/` をGitHub Pagesへデプロイする。
- CDN依存を継続するか、npm依存としてビルドに含めるかをオフライン要件に基づいて決める。
- 古いService Workerから新しいService Workerへ更新できることを実機で確認する。

完了条件: GitHub Pages上で初回表示、更新、再読み込み、オフライン再表示が成功する。

### 6. ヘッダーメニュー改修

- 「新」「削」「コ」を廃止し、ハンバーガーメニュー内の「新規チャット」「全削除」「全コピー」へ移す。
- 設定項目を3ボタン個別表示からハンバーガーボタン表示へ置き換える。
- 「全削除」は先頭項目を残して、それ以外を削除する。
- 既存設定を読み込んだ利用者向けの移行規則を定義する。

完了条件: メニューのキーボード・タッチ操作、外側タップでの閉鎖、設定永続化、削除境界をテストできている。

## 検証項目

- 初回起動と既存IndexedDBからの起動。
- 新規チャット、保存、読込、複製、削除、インポート、エクスポート。
- 設定の保存、再読込、既存設定キーの保持。
- 全APIプロバイダーのリクエスト生成。実API呼び出しを伴わない検証を優先する。
- ストリーミング中断、再送、編集、添付、校正、Twin-engine。
- チャット・履歴・設定の画面遷移とスマートフォン幅での操作。
- Service Workerの初回登録、更新、キャッシュ消去、オフライン再表示。
- ハンバーガーメニューの表示設定と、「全削除」で先頭項目が残ること。

## 想定リスクと対策

- 実行順序の変化: 外部ファイル化の段階では古典スクリプトの順序を維持し、ESモジュール化は別フェーズにする。
- グローバル依存の見落とし: 参照関係を小さな単位で切り、各移動後にスモークテストする。
- IndexedDB互換性の破壊: DBバージョンを移行作業だけでは上げず、保存データの比較テストを行う。
- PWAの古いキャッシュ残留: Service Worker切り替えを独立フェーズにし、旧版からの更新経路を検証する。
- 巨大な移動差分: CSS、JavaScript、TypeScript化、機能変更を別コミットにする。
- GitHub Pagesの公開停止: 新旧の公開方式を並行検証してからPagesの配布元を切り替える。

## コミット方針

原則として、次の単位より大きなコミットを作らない。

1. 計画・ベースライン検証。
2. ビルド設定のみ。
3. CSSの機械的移動のみ。
4. JavaScriptの機械的移動のみ。
5. 型定義と、対応する小範囲のTypeScript化。
6. 1機能または1層のモジュール化。
7. Service Workerとデプロイ設定。
8. ヘッダーメニューの機能変更。

各コミットでビルドと該当する検証を成功させ、問題があればコミット単位で原因を切り分けられるようにする。
